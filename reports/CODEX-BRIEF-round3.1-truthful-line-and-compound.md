# CODEX Brief — Round 3.1: make the code-owned listing line truthful + handle compound turns

**Branch:** `main`
**Context:** Round 3 (code-owned product-listing line) succeeded at its target — the
adversarial hunter shows `contradicts-self` dropped 38 → 13. But it introduced two
regressions, both direct side effects of the code-owned line:
- `scope-loss` 32 → 54
- `ignores-user` 18 → 41

This round fixes those two WITHOUT reverting R3. Do not bring back LLM-written product
facts. The fix is to make the code-owned line faithful and to stop it from suppressing
non-product answers.

---

## REGRESSION 1 — the line states the REQUESTED scope, not the RETURNED scope

Hunter evidence (real, deployed):
- "any in red?" → **"Here are the red women's sneakers I found"** — but the cards are
  terracotta / burgundy / wine (color was semantically relaxed). The line says "red"
  over a non-red pool.
- "sandals, black?" → **"I couldn't find black women's sandals"** — but black sandals
  exist and were shown earlier. The inverse lie.
- "flat … men's sneakers too?" → "Here are the men's sneakers I found" — ignored the
  stated arch constraint.

### Fix
The code-owned line is generated by code, so code must make it match the ACTUAL cards:

- Every attribute the line states (color, gender, category) must be VERIFIED TRUE of the
  shown cards. If it isn't true of (essentially) all shown cards, the line must not
  assert it.
- If a requested filter was relaxed or semantically widened (red → warm tones; exact
  color → nearest), the line must say so honestly: e.g. "I couldn't find an exact red,
  but here are similar warm-toned women's sneakers" — NOT "here are the red sneakers."
  Code already knows the relaxation happened; surface it.
- Never claim "I couldn't find X" when X is in the shown pool. The no-match/relaxation
  wording must be derived from the actual returned set, not the requested filter.

This is the same rule as R3, applied one level deeper: the line is code-owned, so code
must check the line against the cards it is about to render.

---

## REGRESSION 2 — compound turns drop the non-product clause

Hunter evidence:
- "do you guys have any discount codes? also yeah sneakers would work" → "Here's what I
  found for men's sneakers." (discount question dropped)
- "what's your return policy and do you have these in brown?" → answers policy only, no
  brown / no products.
- "the Jillian in brown? also any discount codes?" → "Tell me a bit more…" (answers
  neither)

The code-owned product line is now the WHOLE reply on a listing turn, so when the
customer also asked a non-product question (policy, discount, shipping, general), that
half gets no answer.

### Fix
A turn can have two parts: a product part and a non-product part. They are not mutually
exclusive.
- The code-owned line handles the PRODUCT half (truthfully, per Regression 1).
- The NON-product half (policy / discount / shipping / loyalty / general question) must
  STILL be answered — by the existing policy/support answer path or the LLM, grounded as
  usual.
- Compose them: answer the non-product clause, then present the products with the
  code-owned line. Neither half may silently disappear.

Do not solve this by handing product facts back to the LLM. The product line stays
code-owned; only the non-product clause uses the LLM/policy path.
